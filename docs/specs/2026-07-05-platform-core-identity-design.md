# P1 — Платформенное ядро (Identity + Workspace). Дизайн

**Дата:** 2026-07-05
**Статус:** утверждён к реализации
**Область:** первый под-проект платформенной инициативы (P1). Далее — P2 (миграция
Papyrus в клиента платформы), P3 (Papyrus Lite), P4+ (прочие продукты).

---

## 1. Контекст и стратегические решения

Papyrus превращается из единого приложения в **платформу продуктов с
централизованной идентичностью** (модель Google/Yandex/Atlassian): один аккаунт,
единый вход (SSO), сколько угодно продуктов.

Принятые решения (из брейншторминга):

1. **Коммерческая модель — SaaS, self-serve.** Компания/пользователь заводится сам,
   без модерации платформой.
2. **Перестраиваем фундамент сейчас** (не «когда-нибудь потом»). Тяжёлая СЭД
   (существующий Papyrus) во фризе по новым фичам на время P1–P2.
3. **Интернационализация:** Identity + платформа + Papyrus Lite —
   архитектурно международные (локаль, таймзоны, каналы отправки). Тяжёлая СЭД с
   ЭЦП остаётся **KZ-first** как якорный монетизируемый продукт.
4. **Identity — гибрид:** Ory Hydra (сертифицированный OAuth2/OIDC-движок: токены,
   PKCE, ротация ключей) + наш сервис (логин/consent UI, хранилище пользователей,
   профиль). Security-критичную крипто-часть сами не пишем.
5. **Workspace — платформенный, общий.** Создаётся один раз, виден всем продуктам.
   Модель Slack/Google Workspace.
6. **Граница платформа/продукт:**
   - Платформа владеет: аккаунтами, воркспейсами, участниками, оргструктурой
     (подразделения, должности), общими ролями (owner/admin/member), включением
     продуктов.
   - Продукт владеет своей спецификой (Papyrus: роли док-оборота, доктайпы,
     workflow, ЭЦП, номенклатура, регистрация).
7. **Оргструктура — центральная**, но живёт в Workspace-слое, а НЕ внутри
   OIDC-движка. Держим её общей и простой (дерево юнитов + должности + членство);
   продуктовую специфику наверх не тащим.

## 2. Область (scope)

### Входит в P1 MVP
Критерий MVP: минимум, при котором Papyrus (P2) сможет залогинить пользователя
через OIDC и прочитать команду через REST.

- **Identity:** регистрация, верификация email, логин, logout, сброс пароля,
  сессии с полной информацией + завершение сессии.
- **Hydra-интеграция:** login/consent-хендлеры; trusted-клиенты (авто-consent для
  собственных продуктов).
- **Workspace:** создание, участники, приглашение по email + приём приглашения.
- **Оргструктура:** подразделения (дерево) + должности — базовый CRUD.
- **Products:** реестр продуктов + включение продукта в воркспейсе.
- **REST API для продуктов:** `/me`, `/me/workspaces`,
  `/workspaces/:id/members`, `/workspaces/:id/org-units`,
  `/workspaces/:id/positions`.
- **Аккаунт-хаб UI:** логин/регистрация/consent/профиль/сессии/воркспейсы/свитчер —
  функциональный минимум без полировки.
- **Docker:** весь стек в docker-compose.

### НЕ входит в P1 (в бэклог с пометкой фазы)
Соц-логин (Google и т.д.), phone/OTP, вход по ЭЦП, MFA/2FA, биллинг, гео-IP
«красоты», суб-организации/холдинги, продвинутый RBAC, полировка UI.

## 3. Архитектура

### 3.1 Топология

```
                          ┌─────────────────────────────────────┐
   Браузер / продукты  →  │           ПЛАТФОРМА                  │
                          │  ┌───────────────┐  ┌────────────┐   │
   OAuth2/OIDC потоки  →  │  │  Ory Hydra    │  │ PG (hydra) │   │
   (/authorize,/token,    │  │  токены,      │  └────────────┘   │
    JWKS)                 │  │  сертиф.крипто│                   │
                          │  └──────┬────────┘                   │
                          │         │ login/consent callbacks    │
                          │         ▼                            │
                          │  ┌──────────────────────────────┐    │
   Наш UX аккаунта     →  │  │  Platform Core (Go)          │    │
                          │  │   Identity | Workspace | Products   │
                          │  └──────┬───────────────┬───────┘    │
                          │         ▼               ▼            │
                          │   ┌───────────┐   ┌──────────┐       │
                          │   │ PG (наша) │   │  Redis   │       │
                          │   └───────────┘   └──────────┘       │
                          └─────────────────────────────────────┘
                                        ▲
                          ┌─────────────┴──────────────┐
                          │  Papyrus │ Lite │ Kanban…   │  ← продукты-клиенты
                          └────────────────────────────┘
```

**Три модуля в одном деплое Platform Core** (разные зоны ответственности, единый
сервис — меньше инфры на старте):
- **Identity** — «кто ты»: аккаунт, пароль, email-верификация, login/consent для
  Hydra, профиль, сессии.
- **Workspace** — «кто ты в команде»: воркспейсы, участники, оргструктура, общие роли.
- **Products** — реестр продуктов и их включение в воркспейсе.

Hydra отвечает только за OAuth2/OIDC-механику. Platform Core говорит Hydra «этот
юзер аутентифицирован» и рисует экраны логина/согласия.

### 3.2 Модель данных

**Identity-модуль (наша БД):**
```
users            id, email, email_verified, password_hash,
                 name, avatar_url, locale, timezone, created_at
sessions         id, user_id, hydra_session_id, device_name,
                 user_agent, ip, location, created_at,
                 last_seen_at, ended_at, is_current
```
Одноразовые токены (верификация email, сброс пароля) — **в Redis** с TTL, не в
Postgres. OAuth-клиентов, токены и consent хранит сам Hydra в своей БД.

**Workspace-модуль:**
```
workspaces          id, name, slug, created_by, created_at
workspace_members   workspace_id, user_id, role[owner|admin|member],
                    status[invited|active], org_unit_id?, position_id?
workspace_invites   workspace_id, email, role, token, expires_at, accepted_at
org_units           workspace_id, parent_id, name, sort_order   (дерево)
positions           workspace_id, title
```

**Products-модуль:**
```
products            key[papyrus|lite|kanban...], name
workspace_products  workspace_id, product_id, enabled_at
```

### 3.3 Принцип обмена данными с продуктами
- **Токен «тонкий»:** OIDC-токен несёт только личность — `sub`, email, name, locale.
  Оргструктуры в токене нет.
- **Оргданные — через REST платформы:** продукт с access-токеном дёргает `/me`,
  `/me/workspaces`, `/workspaces/:id/members|org-units|positions`. Один источник
  правды.
- **Контекст воркспейса** выбирает пользователь в продукте (свитчер); продукт
  передаёт `workspace_id` в свои запросы.

## 4. Ключевые сценарии

### 4.1 Регистрация
`/register` → email+пароль+имя → создаём `users` (email_verified=false) → письмо с
токеном → переход по ссылке → email_verified=true → можно логиниться. Модерации нет.

### 4.2 Логин + SSO
Продукт → редирект на Hydra `/authorize` → нет сессии → Hydra зовёт наш `/login` →
проверяем `users` → создаём `sessions` (device/ip/ua/geo) → говорим Hydra
«залогинен» → Hydra выдаёт code → продукт меняет на токены. Повторный вход в любой
продукт → Hydra видит SSO-сессию → без пароля.

### 4.3 Consent
Первый вход продукта → Hydra зовёт наш `/consent` → показываем запрашиваемые
скоупы → соглашаемся → запоминаем. Для собственных продуктов — trusted, авто-пропуск.

### 4.4 Создание воркспейса
Аккаунт-хаб → «Создать воркспейс» → `workspaces` (created_by) → создатель =
`workspace_members` (owner, active).

### 4.5 Приглашение участника
owner/admin → email+роль(+подразделение/должность) → `workspace_invites`
(token, expires) → письмо → получатель: есть аккаунт — принимает; нет —
регистрируется, затем принимает → `workspace_members` (active).

### 4.6 Включение продукта
owner/admin → «Продукты» → включить Papyrus → `workspace_products` → продукт виден
участникам воркспейса.

### 4.7 Управление сессиями
Профиль → «Активные сессии» (список из `sessions`: устройство/где/когда/текущая) →
«Завершить» → отзыв в Hydra (login-сессия + токены) + `ended_at`. «Выйти везде» →
отзыв всех сессий пользователя.

### 4.8 Выход
Logout из продукта → завершаем платформенную сессию → выход из всех продуктов (SSO).

## 5. Стек и раскладка

- **Go** + **Postgres** (наша БД) + **Redis** (кэш сессий, rate-limit на логин,
  одноразовые токены верификации/сброса).
- **Ory Hydra** — OAuth2/OIDC (своя Postgres-БД).
- **Чистая архитектура** + **DI-контейнер** + **TDD**.

**Раскладка проекта:**
```
cmd/server/               точка входа
internal/
  domain/                 сущности, value-objects, порты (repo, hydra, mailer)
  application/            use-cases (RegisterUser, Login, CreateWorkspace,
                          InviteMember, TerminateSession, EnableProduct…)
  infrastructure/         pg-репозитории, redis, hydra-клиент, mailer, DI-wiring,
                          миграции
  presentation/           http: login/consent UX + REST API для продуктов
```

**Конкретные технологии:**
- **DI:** `google/wire` (compile-time, без рантайм-магии; хорошо для clean arch + TDD).
- **HTTP:** `chi`.
- **PG:** `pgx` + `golang-migrate`.
- **Тесты:** `testing` + `testify`; интеграционные — `testcontainers`
  (реальные Postgres/Redis/Hydra в тестах).

**Методы логина в MVP:** только email+пароль + верификация + сброс. Остальное — позже.

## 6. Docker (docker-compose)

```
services:
  platform-core      Go-сервис: login/consent UX + REST API
  hydra              ory/hydra (public :4444, admin :4445)
  hydra-migrate      прогон миграций Hydra (one-shot)
  platform-postgres  Postgres — БД платформы и Hydra (две логические БД в одном
                     инстансе на старте; можно разнести позже)
  redis              кэш сессий, rate-limit, одноразовые токены
```
Все сервисы в одной сети; healthcheck’и на каждом (Hydra — `/health/ready`,
platform-core — свой `/healthz`). Продукты (Papyrus и др.) подключаются к
`platform-core` и Hydra по сети.

## 7. Порядок сборки P1 (TDD, по слоям)

```
1. Каркас: Go clean-arch + DI (wire) + Postgres/Redis + миграции + Hydra в
   docker-compose + healthz
2. Identity: users, регистрация, верификация email, логин, сброс, сессии
   (+ Hydra login/consent-хендлеры, trusted-клиенты)
3. Workspace: воркспейсы, участники, приглашения по email
4. Оргструктура: подразделения (дерево), должности
5. Products: реестр + включение продукта
6. REST API для продуктов (`/me`, `/me/workspaces`, `/workspaces/:id/...`)
7. Аккаунт-хаб UI (функциональный минимум)
```
Каждый шаг — тесты вперёд (TDD): сначала use-case тесты через порты-интерфейсы,
затем инфраструктурная реализация, затем интеграционные через testcontainers.

## 8. Влияние на существующий код (для P2)

P1 самодостаточен и не меняет Papyrus. Но фиксирует, что в P2:
- `companies` → маппятся в `workspaces` платформы;
- `employees` → в `workspace_members` (+ данные личности уезжают в `users`);
- `subdivisions` → в `org_units` платформы; должности — в `positions`;
- auth Papyrus (`sessions`, `refresh_tokens`, `LoginUseCase`) заменяется на OIDC-клиента платформы;
- ролевые enum’ы Papyrus (chief/clerk/…) остаются продуктовой спецификой.
Вчерашний §4.1 STRATEGY.md (регистрация компании + модерация + супер-админ)
**отменён** — заменяется P1 (аккаунт) + воркспейсом в один клик.

## 9. Открытые вопросы (не блокируют P1, решить к P2)
- Как продукт узнаёт «активный воркспейс» между сессиями — хранить last-active на
  платформе или в продукте? (склоняемся к продукту).
- Отдельные ли Postgres-инстансы для платформы и Hydra в проде (в MVP — общий).
- Репозиторий: Platform Core в монорепо или отдельный репо. **РЕШЕНО: монорепо (`platform/`).**
- Энумерация email при регистрации: `POST /register` возвращает 409 на дубль,
  что раскрывает существование аккаунта (в отличие от «тихого» сброса пароля).
  Принятый компромисс self-serve для 2a; смягчить в 2b при вводе HTML-флоу
  (напр. generic-ответ «мы отправили письмо»).
